# MemDB: База данных памяти для ИИ-агентов

> Что это, зачем нужно, чем отличается от upstream MemOS и от конкурентов.

---

## Два проекта, общий предок

**MemOS** (MemTensor) и **MemDB** — это сегодня **два разных проекта** с разной архитектурой, разным стеком и разными целями.

| | MemOS (upstream) | MemDB (форк) |
|---|---|---|
| **GitHub** | [MemTensor/MemOS](https://github.com/MemTensor/MemOS) | [MemDBai/MemDB](https://github.com/MemDBai/MemDB) |
| **Звёзды** | ~5 000 | — |
| **Всего коммитов** | ~966 | 1 066 (966 upstream + **100 своих**) |
| **Основная БД** | Neo4j | PostgreSQL (Apache AGE + pgvector) |
| **Векторная БД преф.** | Milvus | Qdrant |
| **Интернет-поиск** | Bocha API (платный) | SearXNG (self-hosted) |
| **Язык** | Python | Python + **Go** (18K строк) |
| **Деплой** | 3 контейнера (app, neo4j, qdrant) | 10+ контейнеров |
| **Лицензия** | Apache 2.0 | Apache 2.0 |

Общий предок — первый коммит MemOS (`2e6f6e8`, июль 2025). MemDB начал расходиться с февраля 2026 и ушёл в хард-форк 9 февраля 2026 (коммит `243dc5c`, ребрендинг memos -> memdb).

## Часть 1: Что даёт MemOS (upstream)

MemOS — open-source проект MemTensor и нескольких китайских университетов. Академическая основа: статья ["MemOS: A Memory OS for AI System"](https://arxiv.org/abs/2507.03724) (36 страниц, 39 авторов).

### Идея

LLM не имеют состояния. Забывают всё между сессиями. RAG — костыль без управления жизненным циклом, дедупликации, версионирования. MemOS предлагает обращаться с памятью ИИ как с ресурсом ОС — со своим планировщиком, жизненным циклом и хранилищем.

### Пять типов памяти (из статьи)

| Тип | Суть | Хранение |
|-----|------|----------|
| **Долговременная (text_mem)** | Факты, резюме, извлечённые знания. Иерархия: задача -> концепция -> факт | Графовая БД + эмбеддинги |
| **Навыки (skill_mem)** | Процедурные знания — как делать вещи. Поля: name, procedure, experience, examples, scripts | Графовая БД |
| **Пользовательская (user_mem)** | Профиль пользователя, интересы, паттерны поведения | Графовая БД |
| **Параметрическая (para_mem)** | Знания в весах модели — LoRA-адаптеры, доменные патчи | Файловое хранилище |
| **Активационная (act_mem)** | KV-кэш инференса — персистентное когнитивное состояние | Тензоры (vLLM/HuggingFace) |

### Абстракция MemCube

Контейнер для любого типа памяти + метаданные (происхождение, права доступа, TTL, версии). Жизненный цикл: Generated -> Activated -> Merged -> Archived -> Deleted. Кросс-типовая эволюция: текст может стать активационной памятью, стабильные знания — параметрической.

### Что upstream реально даёт из коробки

- **Графовые БД**: Neo4j (основная), Neo4j Community, NebulGraph, PolarDB (один файл 5 672 строки), PostgreSQL (наш PR)
- **Векторные БД**: Qdrant, Milvus
- **Поиск**: Vector, VEC_COT (LLM-декомпозиция запроса), BM25, fulltext (tsvector), интернет (Bocha/Google/Xinyu)
- **LLM**: OpenAI, Ollama, произвольный API
- **Эмбеддинги**: OpenAI, VoyageAI, локальные модели
- **Планировщик**: Redis Streams — асинхронная обработка add/query/feedback
- **Реорганизатор**: LLM находит конфликты между записями (cosine > 0.8) -> merge/archive
- **Память навыков**: LLM разбивает разговор на задачи, извлекает структурированные навыки
- **Преференции**: Явные + неявные, извлекаются LLM из разговоров
- **MCP-сервер**: Model Context Protocol для интеграции с ИИ-клиентами
- **Деплой**: Docker Compose (3 контейнера: app + Neo4j + Qdrant)

### Чего upstream НЕ даёт

- Go-шлюз (нет)
- Русский язык в токенизаторе (только EN/ZH)
- Дедупликация перед вставкой (нет — только реорганизатор после)
- Кросс-клиентская хэш-дедупликация (нет)
- Реальный MMR-дедуп (параметр есть, но `dedup="mmr"` = обычный текстовый дедуп)
- SearXNG (вместо этого — платный Bocha API)
- Разложение PolarDB в пакет (монолит 5 672 строки)

## Часть 2: Что MemDB добавил сверху

100 собственных коммитов. Вот конкретика — что есть в MemDB и чего нет в upstream.

### A. Go-шлюз (memdb-go) — 18K строк, с нуля

Целый сервис на Go, которого нет в upstream ни в каком виде.

```
Клиент -> Go (:8080) --нативно--> PolarDB / Qdrant / Redis
                      --прокси---> Python (:8000) -> LLM
```

**15 из 33 маршрутов нативные** (45%):
- `/product/search` — полный пайплайн: VoyageAI embed -> параллельный errgroup (PolarDB вектор + PolarDB полнотекст + Qdrant преференции) -> merge -> dedup -> ответ
- `/product/get_all` — пагинированное чтение всей памяти
- `/product/delete_memory`, `/product/get_memory`, `/product/users/*`, `/health`, и т.д.

**Производительность**:

| Операция | Python (upstream) | Go (MemDB) | Ускорение |
|----------|-------------------|------------|-----------|
| Получить всю память | 25 600мс | 363мс | **70x** |
| Поиск (без кэша) | ~30 000мс | 350-500мс | **60-85x** |
| Поиск (из Redis-кэша) | — | 1.7мс | — |
| Health check | 1.80мс | 0.56мс | **3.2x** |

**18 маршрутов проксируются** на Python (те, что требуют LLM): add, chat/*, feedback, suggestions, scheduler.

**Фоллбэк**: если Go не может обработать (нет эмбеддера, нет БД, fine-режим) — прозрачно проксирует на Python.

### B. PolarDB вместо Neo4j — другой стек

Upstream использует **Neo4j** как основную графовую БД. MemDB заменил на **PolarDB** (Apache AGE на PostgreSQL):
- Граф (Cypher) + вектор (pgvector) + полнотекст (tsvector/GIN) — **всё в одной БД**
- Не нужен отдельный кластер Neo4j
- Upstream имеет `polardb.py` как монолит на 5 672 строки. MemDB **разложил его в пакет** из 11 файлов с миксинами (connection, schema, nodes, edges, search, filters и т.д.)

### C. Qdrant вместо Milvus — для преференций

Upstream использует **Milvus** для преференциальной памяти. MemDB заменил на **Qdrant** через адаптер `qdrant_multi.py`:
- Две коллекции: `explicit_preference`, `implicit_preference`
- Адаптер конвертирует Milvus-формат фильтров в Qdrant-формат
- Не нужен ещё один сервис (Milvus) — Qdrant уже используется

### D. SearXNG вместо Bocha API — self-hosted интернет-поиск

Upstream использует **Bocha API** (платный). MemDB заменил на **SearXNG** (self-hosted метапоиск):
- Агрегирует Google, DuckDuckGo, Brave, Startpage, Wikipedia, GitHub
- Батч-эмбеддинг: все результаты в 1 вызов VoyageAI (upstream делал 30 отдельных вызовов)
- Никаких платных API-ключей для поиска

### E. Русский язык в токенизаторе

Upstream: только EN/ZH. Кириллица тихо теряется в regex `\b[a-zA-Z0-9]+\b`.

MemDB добавил:
- `_is_cyrillic()` — детекция > 20% кириллицы
- `_tokenize_cyrillic()` — regex `[a-zA-ZА-Яа-яёЁ0-9]+`
- ~60 русских стоп-слов
- `detect_lang()` возвращает `"ru"`
- Исправлен английский regex — теперь не теряет случайные кириллические токены

Это влияет на BM25, fulltext и VEC_COT — все стратегии поиска работают с русским.

### F. Реальный MMR-дедуп

В upstream `dedup="mmr"` — это **фикция**: тот же текстовый дедуп, что и `"sim"`.

MemDB реализовал настоящий MMR:
- **Python**: Эмбеддинг всех результатов -> матрица косинусного сходства -> `find_best_unrelated_subgroup(bar=0.8)`
- **Go**: 3-фазный алгоритм — предзаполнение top-2, MMR lambda=0.8 с экспоненциальным штрафом, пересортировка по релевантности
- Фоллбэк на текстовый дедуп при ошибке

### G. Косинусная дедупликация при вставке

Upstream не проверяет дубликаты перед вставкой (только реорганизатор LLM *после*).

MemDB добавил `_deduplicate_memories()`:
- Перед вставкой: `search_by_embedding(threshold=0.95)` — ищет похожие активные записи
- Порог 0.95 (нормализованный) = 0.90 (сырой косинус)
- Разные пороги: fast-режим 0.92, fine-режим 0.95

### H. Хэш-дедупликация (кросс-клиентская)

Upstream не имеет. MemDB добавил:
- Клиенты генерируют `SHA256[:16]` хэш контента в `info.content_hash`
- Перед косинусной проверкой — точное совпадение хэша через `get_by_metadata()`
- Работает между Claude Code, OpenClaw и API — один и тот же факт не сохранится дважды

### I. tsvector в схеме PolarDB

Upstream: fulltext search существует, но колонка `properties_tsvector_zh` **не создаётся автоматически** при инициализации БД.

MemDB: добавил в `schema.py:_create_graph()`:
- Колонка `properties_tsvector_zh tsvector`
- Триггер `update_tsvector_zh()` — автообновление при INSERT/UPDATE
- GIN-индекс `idx_memory_tsvector_zh`
- Бэкфил существующих NULL-строк

Fulltext поиск теперь переживает пересоздание БД.

### J. Десятки багфиксов, которых нет в upstream

- `parse_json_result(None)` — VEC_COT крэш при пустом ответе LLM
- `cosine_similarity_matrix` — numpy крэш на None эмбеддингах
- `agtype_object_field_text(jsonb)` — AGE-функция на JSONB-колонке
- `status` как kwarg в PolarDB `get_by_metadata` — перенесён в filter list
- `jiebacfg` tsquery config → `simple` (нет pg_jieba расширения)
- MCP-сервер крэш при инициализации эмбеддера
- OSS опциональный для Skill Memory
- KV-кэш крэш при отсутствии локальной модели
- AuthConfig warning spam
- И ещё ~30 фиксов

## Часть 3: Что upstream добавил, а MemDB не взял

MemOS v2.0.5 (10 февраля 2026) содержит фичи, которые MemDB **сознательно не мержил**:

| Фича upstream v2.0.5 | Почему не взяли |
|----------------------|-----------------|
| **Рефакторинг планировщика** (#1004) — модуляризация handlers/pipelines | Конфликты с нашей декомпозицией PolarDB + кастомными retriever'ами |
| **Auth middleware** — SHA-256 API ключи, scopes | В Go-шлюзе своя авторизация (master key + rate limiting) |
| **Rate limiting middleware** — Redis sliding window | Реализовано в Go-шлюзе |
| **RawFileMemory** — recall оригинального текста из KB | Не нужно для нашего use case |
| **Xinyu search** — китайский поисковик | Заменён SearXNG |

### Cherry-picks, которые взяли

- **#999**: Автоизвлечение ключевых слов из `metadata.key` в `tags` (улучшает BM25/fulltext)
- **#1006**: `relativity` threshold, text similarity helpers (dice/tfidf/bigram)
- **#1038**: URL protection, markdown header fix

### Честное сравнение с upstream MemOS

MemDB и MemOS решают одну задачу, но **по-разному**:

| Аспект | MemOS лучше | MemDB лучше |
|--------|-------------|-------------|
| **Экосистема** | 5k звёзд, 30 контрибьюторов, еженедельные релизы | Частный форк, 1 разработчик |
| **Графовая БД** | Neo4j — зрелая графовая БД, Cypher | PostgreSQL AGE — всё в одной БД, но AGE менее зрелый |
| **Auth** | Полноценный middleware с API-ключами и scopes | Базовый master key в Go |
| **Upstream фичи** | Рефакторинг планировщика, RawFileMemory, auth | — |
| **Производительность** | — | **В 70 раз быстрее** (Go-шлюз) |
| **Качество поиска** | MMR — фикция, нет дедупликации при вставке | **Реальный MMR, 4-уровневая дедупликация** |
| **Русский язык** | Нет | **Полная поддержка** |
| **Операционная простота** | 3 контейнера (но нужен Neo4j) | 10+ контейнеров (но всё на PostgreSQL) |
| **Зависимости** | Milvus, Bocha (платный) | Всё self-hosted, без платных API |

          |                        |

          v                        v

## Итог

**MemOS** — академический open-source проект с сильной теоретической базой (73.31% на LoCoMo, 5 типов памяти, статья на 36 страниц). Даёт ядро: извлечение навыков, реорганизатор, планировщик, VEC_COT, BM25.

**MemDB** — продакшен-форк, который взял это ядро и:
- Построил Go-шлюз с **70x ускорением** для операций чтения
- Заменил Neo4j+Milvus+Bocha на **PostgreSQL+Qdrant+SearXNG** (единый self-hosted стек)
- Добавил **реальный MMR-дедуп** вместо upstream-фикции
- Добавил **4-уровневую дедупликацию** вместо zero-dedup-before-insert
- Добавил **русский язык** в токенизатор
- Исправил **~30 багов** в Python-ядре
- Разложил монолитный PolarDB (5 672 строки) в пакет из 11 файлов

100 коммитов, 18K строк Go, десятки исправлений Python — это уже другой проект с другой архитектурой.
