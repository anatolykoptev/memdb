# MemDB v0.23.0 — M10 User Profiles + Performance + Security Audit

**2026-04-26**

MemDB v0.23.0 ships the **M10 sprint**: a structured `user_profiles` layer
(Memobase moat), L1/L2/L3 memory layer API, Helm chart, cross-encoder
pre-compute, PageRank background scheduler, reward scaffold, plus a five-finding
security audit (cube-isolation, prompt injection, DoS admission control,
multi-replica advisory lock, search cache key correctness). Headline:
**MemDB scores 72.5% LLM Judge** on LoCoMo chat-50 (excl cat-5),
between MemOS (73.31%) and Zep (75.14%), +5.62pp ahead of Mem0 (66.88%),
-2.64pp short of Zep, -3.28pp short of Memobase (75.78%) leader,
and +2.5pp vs the v0.22.0 result of 70.0%.

## Security & correctness

The whole M10 stack went through a five-finding internal security audit before
release. All issues were patched in this release.

| Finding | Severity | PR | Fix |
|---|---|---|---|
| **C1** `user_profiles` missing `cube_id` → cross-tenant leak | High | [#129](https://github.com/anatolykoptev/memdb/pull/129) | Migration `0017` adds `cube_id` column + cube-scoped unique index `(cube_id, user_id, topic, sub_topic)`. Lookup goes through `GetProfilesByUserCube` (NULL legacy rows are never returned). |
| **C2** Profile facts can carry prompt-injection payloads into chat system prompt | High | [#130](https://github.com/anatolykoptev/memdb/pull/130) | Sanitize control characters + tag-wrap each fact in `<fact>...</fact>`; LLM is instructed to treat fact contents as data, never instruction. |
| **C3** Profile-extract goroutine had no admission control → DoS via burst | Medium | [#131](https://github.com/anatolykoptev/memdb/pull/131) | Bounded semaphore (size 8) acquired BEFORE goroutine spawn. Saturated calls drop with a `busy` outcome counter. |
| **I4** PageRank background job ran on every replica → wasted CPU + race writes | Medium | [#132](https://github.com/anatolykoptev/memdb/pull/132) | Postgres advisory lock around the compute phase; first replica wins, others skip. |
| **P3** Search cache key didn't include `level` / `agent_id` / `pref_top_k` → wrong-result returns under load | Medium | [#133](https://github.com/anatolykoptev/memdb/pull/133) | Cache key now includes the missing fields; correctness verified via a unit-level diff test. |

- **Query phase: ~10 hours** with `--workers 4` outer parallelism +
  `D2_MAX_HOP=2`. Outer parallelism is new in this release
  (`evaluation/locomo/query.py --workers N`, default 1 sequential).

## Migration

### Schema

Three new auto-applied migrations (run on `memdb-go` startup via
`postgres_migrations.go`):

| ID | Purpose |
|---|---|
| `0015_user_profiles_table.sql` | New `memos_graph.user_profiles(topic, sub_topic, memo, ...)` table for the structured profile layer. |
| `0016_feedback_events.sql` | `feedback_events` + `extract_examples` tables for the M11 reward loop scaffold. |
| `0017_user_profiles_add_cube_id.sql` | Adds `cube_id TEXT` (nullable for backfill) + cube-scoped unique index on `user_profiles`. Audit C1 fix. |

### `cube_id` backfill on existing `user_profiles` rows

`0017` adds `cube_id` as **nullable** and leaves existing rows with `cube_id =
NULL`. The application getter (`GetProfilesByUserCube`) never returns
NULL-cube rows. This is safe **because production has no benchmark-scale
profile data yet** — only test rows from M10 stream-2 development. A
follow-up migration (`0018`, scheduled for M11) will reap NULL rows and flip
the column to `NOT NULL`.

If you have your own production `user_profiles` rows from a self-hosted
v0.22.0+ build, manually backfill `cube_id` before `0018` lands.

### Wire format

No changes. REST API, MCP tool surface, and on-disk encoding are identical to
v0.22.0.

## Env gate reference

All M10 features are **on by default**. Toggle off for ablation / debugging:

| Variable | Default | Effect when off |
|---|---|---|
| `MEMDB_PROFILE_EXTRACT` | `true` | Skip the LLM profile-extraction hook on `add_fine`. New chats get no fresh profile rows. |
| `MEMDB_PROFILE_INJECT` | `true` | Drop the `<user_profile>` section from chat system prompts. Falls back to memory-only context. |
| `MEMDB_CE_PRECOMPUTE` | `true` | Skip CE pre-compute at D3 ingest; query-time CE call comes back. +100-400ms p95 chat. |
| `MEMDB_PAGERANK_ENABLED` | `true` | Disable PageRank background goroutine. D1 boost gracefully treats missing PageRank as 0. |
| `MEMDB_PAGERANK_INTERVAL` | `6h` | How often the PageRank job runs. Go duration string. |
| `MEMDB_PAGERANK_BOOST_WEIGHT` | `0.1` | Multiplier weight on the PageRank score in D1 rerank. Range `[0, 1]`. |

The `level=l1|l2|l3` query parameter from S4 has no env gate — it's a pure
API addition. Omitting it preserves v0.22.0 behaviour (search across all
layers).

## What's NOT in this release

- **Stretch goal of 78% LLM Judge** — deferred to M11. Closing the remaining
  -3.28pp vs Memobase needs deeper profile coverage (we extract less per
  conversation than Memobase) and cat-2 multi-hop work.
- **Full corpus cat-2 multi-hop** still measures only **29.0%** LLM Judge —
  the D2 BFS expansion is shipped but the recall ceiling on the full 1986
  corpus is still cosine-bound. Dedicated D2 work is a top M11 candidate.
- **`feedback_events` reads** — the table and write paths ship in S8, but no
  retrieval / reward path consumes it yet. Wiring is M11 (closed loop).
- **PageRank advisory-lock observability** — first-replica wins, others skip
  silently. Adding a per-replica `pagerank_skipped_total` counter is a
  follow-up.
- **NULL `cube_id` reaper** for the legacy `user_profiles` rows. Migration
  `0018` will land in M11.

## M11 candidates

Post-release backlog seeded from this sprint:

| Item | Why |
|---|---|
| **Parallelize CE precompute at D3 reorganizer** | Currently per-memory sequential. Worker pool of 4 should halve the D3 phase wall-time on cold ingest. |
| **`COPY FROM` bulk inserts for `memory_edges` + `entity_nodes`** | The 7.5× ingest speedup leaves AGE-Cypher inserts as the next bottleneck. Direct text-format COPY can win another 2-3× on Stage 3 scale. |
| **GIN index on `Memory.properties->'ce_score_topk'`** | The S6 lookup is currently btree on graphid. A GIN expression index over the JSONB key would make the merge step O(log N) at scale. |
| **Semantic prompt-injection classifier** | C2 uses tag-wrapping + sanitization, which catches structural attacks. A small LLM classifier (or a regex bank) would catch semantic payloads ("ignore previous instructions") embedded inside a benign-looking memo. |
| **Close the reward loop (S8 reads)** | Wire `feedback_events` into D1 importance scoring + extract-prompt example bank. Targets the cat-3 (preference) gap. |
| **D2 BFS recall lift for cat-2** | Full-corpus cat-2 LLM Judge is 29% — biggest remaining quality gap. Hub-and-spoke topology in D3 + tuned hop-decay is the path. |
